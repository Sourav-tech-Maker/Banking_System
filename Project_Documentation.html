1.1 Description of the Software System under Study: YONO Bank (You Only Need One) is a production-grade, full-stack digital banking application designed to provide secure and comprehensive online banking services. The proposed software system aims to deliver a seamless digital banking experience encompassing user authentication, account management, fund transfers, KYC verification, beneficiary management, savings goals tracking, and administrative controls. The intended operations include user registration and verification, opening savings and current accounts, performing ACID-compliant fund transfers with double-entry ledger bookkeeping, KYC document submission and review, beneficiary management with OTP verification, savings goal creation and funding, and comprehensive admin panel operations. The system involves the following users with their respective roles:

  • Customer: End-user who registers, manages accounts, performs transactions, submits KYC documents, adds beneficiaries, and tracks savings goals.
  • Admin: Privileged user who reviews KYC applications, manages account statuses (freeze/close/reactivate), reverses transactions, and monitors system-wide statistics.

1.5 Methodology:

1.5.1  SDLC Model to be used:
The Agile (Iterative) Model has been adopted for the development of the YONO Banking System.

1.5.2  Justification for the Selection of Model:
The Agile (Iterative) Model is selected for the following reasons:

  1. Incremental Development: The banking system comprises multiple modules (Authentication, Accounts, Transactions, KYC, Beneficiary, Goals, Dashboard, Admin) which can be developed and tested incrementally in sprints. Each sprint delivers a working module that can be independently tested and reviewed.
  2. Flexibility in Requirements: Banking requirements often evolve during development. The Agile model allows for requirement modifications between sprints without disrupting the entire project timeline. For instance, the addition of the savings goals module and admin transaction reversal feature were incorporated as enhancements in later sprints.
  3. Continuous Integration & Feedback: Each iteration produces a working prototype that can be demonstrated to the project guide, enabling early detection of issues and course corrections. The double-entry ledger system, for example, was refined across multiple iterations based on testing feedback.
  4. Risk Mitigation: Critical modules like the ACID-compliant transaction system and JWT dual-token authentication were developed and tested early in the project lifecycle, ensuring that the most complex and risk-prone components received adequate attention.
  5. Parallel Development: The MERN stack architecture supports parallel development of frontend and backend components. While backend APIs were being developed, frontend UI components were simultaneously built using mock data, significantly reducing the overall development time.

Chapter-2: Software Requirement Specification (SRS)

2.1 Description of Information System (Block Diagram): The YONO Banking System is a comprehensive digital banking platform that manages user accounts, financial transactions, KYC verification, and administrative operations. The functioning of the information system is described below in narrative form in detail. The system operates on a layered architecture pattern with strict separation of concerns. The Client Layer (React 19 SPA) communicates with the API Gateway Layer (Express 5.2) via HTTPS REST JSON requests with HttpOnly cookies. The API Gateway routes requests through the Core Engine (Middleware → Controllers → Services) which interacts with the Data & Infrastructure layer (MongoDB Atlas, ImageKit CDN, Gmail SMTP).

Fig 2.1 Block Diagram for YONO Banking Information System

2.1.1  Product Features: The following major processes have been identified in the YONO Banking System:

  1. 'User Registration & Authentication' – Handles user sign-up, email OTP verification, login with brute-force protection, JWT dual-token rotation, Google OAuth 2.0 sign-in, and secure logout with token blacklisting.
  2. 'KYC Verification' – Manages digital KYC document submission via ImageKit CDN, admin review pipeline (Approve/Reject/Re-submit), and KYC status tracking.
  3. 'Account Management' – Handles opening of Savings & Current accounts (gated by KYC approval), account lifecycle management (Active → Frozen → Closed), and balance computation via ledger aggregation.
  4. 'Transaction Processing' – Core process implementing ACID-compliant fund transfers using MongoDB transaction sessions, idempotent transactions with unique idempotencyKey, and double-entry immutable ledger entries.
  5. 'Beneficiary Management' – Manages OTP-verified beneficiary registration workflow, beneficiary listing, and validated transfer targets.
  6. 'Savings Goals' – Handles goal creation, funding from linked accounts, and progress tracking with visual analytics.
  7. 'Dashboard & Analytics' – Aggregation pipelines for spending charts, statistics cards, recent transactions, and AI-powered financial insights.
  8. 'Admin Panel' – Full admin control center for KYC review, account status management, transaction reversals with ledger audit trail, and system-wide metrics monitoring.
  9. 'Email Notification Service' – Nodemailer-based transactional email delivery for OTPs, security alerts, device detection notifications, and account status updates.
  10. 'Report Generation' – Transaction history reports with pagination, spending analytics, and downloadable account statements.

2.1.3  Procedures/rules/ mathematical relationships: The following procedures, rules, and mathematical relationships have been identified for the YONO Banking System:

(i) Balance Computation (Double-Entry Ledger): The account balance is never stored as a static field. Instead, it is computed in real-time by aggregating all immutable ledger entries associated with the account: Account Balance = Σ (Credit Entries) − Σ (Debit Entries). This ensures that the balance is always consistent with the ledger history and eliminates the possibility of balance discrepancies due to race conditions.

(ii) Transaction Processing Rules (ACID Compliance): Each fund transfer must create exactly two ledger entries: one DEBIT entry for the sender and one CREDIT entry for the receiver. The entire transfer operation is wrapped in a MongoDB transaction session. If any step fails, the entire operation is rolled back (atomicity). Before processing, the system checks: Sender's Balance ≥ Transfer Amount. If insufficient, the transaction is rejected with a 400 status. Idempotency Rule: Each transaction must include a unique idempotencyKey. If a duplicate key is detected, the cached result is returned instead of processing a new transfer.

(iii) Password Hashing Policy: User passwords are hashed using bcryptjs with a configurable number of salt rounds before storage. The original password is never stored in the database. During login, the submitted password is compared against the stored hash using a constant-time comparison function.

(iv) OTP Generation & Verification: OTPs are generated as 6-digit random numbers. Before storage, the OTP is hashed using SHA-256 algorithm. The hashed OTP is stored in the database with a TTL (Time-To-Live) of 10 minutes. During verification, the submitted OTP is hashed and compared against the stored hash.

(v) JWT Token Lifecycle:
• Access Token: Signed with JWT_SECRET, expiry = 15 minutes, stored in volatile memory (React state).
• Refresh Token: Signed with JWT_SECRET, expiry = 7 days, transported via HttpOnly Secure cookies.
• Rotation Rule: On each token refresh, the old refresh token is blacklisted in the database, and a new pair of tokens is issued.

(vi) Brute-Force Protection Policy: If Login Attempts ≥ 5 within a session → Account is locked for 30 minutes. The loginAttempts counter is reset upon successful login.

(vii) Savings Goal Progress Calculation: Progress (%) = (Current Funded Amount / Target Amount) × 100.

(viii) Discount / Fee Policy: Currently, the system does not charge transaction fees for inter-account transfers. However, the architecture supports the addition of fee calculation rules in future iterations.

2.3 Software Product constraints: The following limiting constraints have been identified for developing the YONO Banking System:

  1. MongoDB Atlas (Cloud Database): The system relies on MongoDB Atlas as its cloud-hosted database. The free-tier (M0) cluster has limitations on storage (512 MB), connections (500), and does not support certain enterprise features like sharding. For production deployment, a paid tier (M10 or higher) shall be required, which is a constraint for the software.
  2. ImageKit CDN: The KYC document upload functionality depends on ImageKit's CDN service for storing and serving document images. ImageKit's free plan has a bandwidth limit of 20 GB per month and 20,000 file transformations. For a large-scale deployment, a paid ImageKit plan shall be a constraint for the software.
  3. Gmail OAuth2 SMTP: The email notification service uses Gmail's OAuth2 SMTP for sending transactional emails (OTPs, alerts, notifications). Gmail imposes a sending limit of 500 emails per day for regular accounts and 2,000 for Google Workspace accounts. For high-volume email delivery, a dedicated email service provider (e.g., SendGrid, AWS SES) would be required.
  4. JWT Secret Key Management: The security of the entire authentication system depends on the secrecy of the JWT_SECRET environment variable. If compromised, all tokens become vulnerable. A dedicated secrets management service (e.g., AWS Secrets Manager, HashiCorp Vault) would be required for production.
  5. Google OAuth 2.0 Credentials: The Google Sign-In functionality requires valid OAuth 2.0 credentials (Client ID and Client Secret) from the Google Cloud Console. These credentials are project-specific and must be configured with appropriate redirect URIs. The Google Cloud Console imposes usage quotas on OAuth API calls.
  6. Node.js Runtime: The backend is built exclusively on Node.js 22.x. This is a single-threaded runtime, which may not be optimal for CPU-intensive operations. For handling heavy computational loads, additional infrastructure (worker threads, load balancers) would be required.
  7. CORS Policy: The frontend and backend are deployed on different origins (ports 5173 and 3000 respectively). Cross-Origin Resource Sharing (CORS) must be properly configured to allow cookie-based authentication across origins. Incorrect CORS configuration can break the entire authentication flow.
  8. Browser Compatibility: The frontend uses modern JavaScript features (ES2020+) and React 19. These technologies require modern browsers (Chrome 90+, Firefox 88+, Edge 90+, Safari 15+). Older browsers and Internet Explorer are not supported.
  9. Internet Connectivity: Since the system relies on MongoDB Atlas (cloud database), ImageKit CDN, Gmail SMTP, and Google OAuth, a stable internet connection is mandatory for the application to function. The system cannot operate in offline mode.
  10. i18next Plugins: The internationalization feature uses i18next with plugins (react-i18next, i18next-http-backend, i18next-browser-languagedetector) for multi-language support (EN, ES, FR, DE, HI). These plugins need to be properly configured and translation files must be maintained for all supported languages, which adds an ongoing maintenance constraint.
